# Алгоритм предоценки и фильтрации пар для сокращения людского труда

> Полный план: автоматическая предварительная оценка изображений →
> классификация пар по неопределённости → отправка людям только
> «сомнительных» пар → валидация результатом сравнения с людскими
> оценками и расчётом корреляции.

---

## 1. Постановка задачи

**Дано:**
- N = 1000 медицинских изображений
- 4 критерия оценки: общее качество, резкость, контрастность, артефакты
- Полный набор уникальных пар: `N(N−1)/2 = 499 500` для каждого критерия

**Текущее состояние:**
- Пары выбираются случайно
- Каждая пара требует субъективной оценки человеком
- 500 000 сравнений × 4 критерия = **2 миллиона** действий человека

**Цель:**
1. Автоматически предсказать рейтинг каждого изображения по каждому
   критерию (без человеческой разметки).
2. Использовать предсказание, чтобы **отфильтровать пары**, в которых
   разница автоматических оценок очевидна — их человек оценивать не
   будет.
3. Отправить оценщикам только пары с **близкими** автоматическими
   оценками (где алгоритм не уверен).
4. Валидировать качество фильтрации: дать оценщикам выборки пар из
   разных «зон неопределённости» и посчитать корреляцию.

**Целевое сокращение:** с ~500 000 до **5 000–15 000** ручных
сравнений на критерий (в 30–100 раз меньше).

---

## 2. Идея алгоритма в четырёх фразах

1. Каждому изображению вычисляются объективные метрики качества (без
   эталона) — отдельный набор для каждого критерия. Из метрик
   собирается единое число `μ_i` — предварительный рейтинг
   (нормализация — по умолчанию percentile rank, см. 3.6).
2. Для каждой пары `(i, j)` считается величина `Δ = |μ_i − μ_j|`.
   Если `Δ` велика — алгоритм решает сам; если мала — пара отдаётся
   человеку. Часть бюджета (≈ 7%) **принудительно** отдаётся людям на
   пары, соединяющие разные квантильные кластеры по `μ` — это
   гарантия, что граф человеческих сравнений останется связным
   (см. 5.5).
3. После сбора людских оценок строится финальный рейтинг по
   Брэдли-Терри (Режим A — только людские, для валидации; Режим B —
   с информативным приором из `μ`, для production).
4. Считается корреляция между `μ` и финальным рейтингом из Режима A
   по каждому из трёх диапазонов `Δ`. Если корреляция в зоне
   «алгоритм уверен» высокая — фильтрация надёжна; если низкая —
   пороги пересматриваются или фильтр для критерия отключается.

---

## 3. Этап 1. Автоматическая предоценка

Используется **No-Reference Image Quality Assessment (NR-IQA)** — не
требует «эталонного» изображения. Подбираются метрики, специфичные
для каждого из четырёх критериев.

### 3.0. Калибровочный пилот (обязательный шаг)

Прежде чем считать метрики на всех 1000 изображениях, нужен
**пилот на 30–50 снимках**:

1. **Распределение значений каждой метрики.** Гистограмма для
   проверки нормальности (тест Шапиро-Уилка). Если распределение
   сильно скошено или бимодально → выбираем percentile rank вместо
   z-score (см. 3.6).
2. **Корреляция метрик внутри критерия.** Если две метрики дают
   почти идентичные значения (Pearson > 0.95) — оставляем одну.
3. **Sanity-check глазами.** Отсортировать топ-5 и низ-5 по каждой
   метрике. Просмотреть: соответствует ли «лучшая по NIQE»
   интуитивному впечатлению? Если NIMA ставит одинаково низкие
   баллы всем рентген-снимкам или CLIP-IQA не различает
   медицинские снимки между собой — это прямой сигнал о domain
   shift, и для этого критерия zero-shot модель не годится.
4. **Скорость инференса.** NIQE на 30 изображениях должен
   занимать < 1 мин на CPU; нейросетевые (NIMA, MUSIQ, CLIP-IQA) —
   < 5 мин. Если медленно — переход на batch-инференс на GPU или
   замена модели.

**Без пилота все пороги (`τ_high`, `τ_low`) — гадание.**

Если хотя бы одна метрика по критерию даёт ожидаемое ранжирование —
критерий пригоден к фильтрации. Если ни одна не работает — фильтр для
этого критерия отключается, остаётся только сбор людских голосов.

> [!IMPORTANT]
> До пилота нужно **зафиксировать модальность данных** (рентген, КТ,
> МРТ, гистология, УЗИ). Метрики и пороги, оптимальные для рентгена,
> могут давать обратные результаты на гистологии. Ниже все
> рекомендации — отправная точка, требующая калибровки под конкретный
> датасет.

### 3.1. Резкость

**Что хотим измерить:** насколько чёткие границы и мелкие детали.

**Метрики:**
- **Дисперсия лапласиана** `Var(∇²I)` — самая простая и быстрая.
  Размытое изображение даёт почти нулевой второй пространственный
  градиент. Pech-Pacheco et al. (2000), оригинально для автофокуса
  микроскопа, до сих пор работает.
- **Tenengrad** — сумма квадратов модулей градиента Собеля.
- Альтернатива: компонент `f1` из BRISQUE (Mittal et al. 2012),
  отвечающий за статистику высокочастотной информации.

**Рекомендация:** дисперсия лапласиана + Tenengrad, среднее их
z-нормализованных значений.

### 3.2. Контрастность

**Что хотим измерить:** разнообразие интенсивностей.

**Метрики:**
- **RMS-контраст** = `σ(I) / mean(I)` — стандартное отклонение
  яркостей делённое на среднее. Базовая мера разброса.
- **Контраст Майкельсона** = `(I_max − I_min) / (I_max + I_min)` —
  для бинарных или сильно-полярных изображений.
- **Энтропия гистограммы** `H = −Σ p_i log p_i` — сколько информации
  в распределении интенсивностей.

**Рекомендация:** RMS-контраст + энтропия (среднее z-нормализованных).

### 3.3. Артефакты

**Что хотим измерить:** шум, блокинг (после JPEG), полосы, искажения.

**Метрики:**
- **NIQE** (Mittal et al. 2013) — «полностью слепая» метрика.
  Сравнивает локальные статистики изображения с моделью «чистых»
  снимков. Не требует обучения. **Меньше = лучше.**
- **PIQE** (Venkatanath et al. 2015) — оценивает искажения по
  блокам.
- Оценка шума по высоким частотам вейвлет-разложения (метод
  медианного абсолютного отклонения детализированного канала; Donoho
  & Johnstone 1994).

**Рекомендация:** NIQE как основная метрика; вспомогательно — оценка
шума через вейвлеты. Оба инвертируем (бóльшее значение → выше
качество), затем z-нормализуем.

### 3.4. Общее качество

**Что хотим измерить:** интегральная оценка восприятия.

**Метрики:**
- **NIMA** (Talebi & Milanfar 2018) — нейросеть, обученная на
  AVA dataset. Возвращает распределение оценок (1–10), используем
  среднее как скалярный рейтинг.
- **MUSIQ** (Ke et al. 2021) — мульти-масштабный трансформер.
  Точнее NIMA, но тяжелее.
- **CLIP-IQA** (Wang et al. 2023) — использует zero-shot
  возможности CLIP. Удобно: не требует тренировки.
- Эвристика-fallback: взвешенная сумма z-нормализованных рейтингов
  по другим трём критериям с весами, подобранными по первой партии
  человеческих данных.

**Рекомендация:** **не полагаться на zero-shot модели для общего
качества дольше первого пилота.** На медицинских данных они часто
ведут себя катастрофически: NIMA, обученная на AVA (любительские
фотографии), может ставить одинаково низкие баллы всем рентгенам
из-за того, что они «чёрно-белые скучные»; CLIP-IQA вообще не имеет
понятия о специфике патологии и иногда трактует диагностически
важные особенности как «артефакт».

**Двухэтапный путь:**

1. **Старт (zero-shot):** CLIP-IQA или NIMA как заглушка, чтобы
   запустить пилот и собрать первые ~500–1000 человеческих
   сравнений. На этом этапе общее качество может работать плохо —
   это нормально.
2. **Перевод на fine-tuning через 500–1000 голосов:**
   - Извлечь признаки замороженной CNN (EfficientNet-B0 или
     ResNet-50, веса ImageNet) → вектор размерности ~1280.
   - Обучить лёгкую голову (логистическая регрессия / 2-слойный MLP)
     с **margin ranking loss**:
     `L = max(0, m − sign · (s_A − s_B))` (Burges et al. 2005).
   - На 500 парах достигается стабильный регрессор за 5 минут на
     CPU.
   - Финальный `μ_overall` — выход обученной головы, нормализованный.

**Альтернатива:** взвешенная сумма z-нормализованных рейтингов по
другим трём критериям с весами, подобранными по первой партии
человеческих данных (через regression на парах). См. также
оговорку о нелинейности в 3.5.

### 3.5. Сводный «предрейтинг» μ_i

Для критерия `c` и изображения `i`:

```
μ_i^(c) = normalize( combine(metrics_for_c) )
```

**Combine** — способ объединения метрик внутри критерия. Для
резкости/контраста простая линейная комбинация даёт хороший
результат: соответствующие метрики измеряют примерно одно и то же
свойство, и их среднее устойчивее каждой по отдельности.

Для критерия «общее качество» **линейная сумма опасна**: связь между
резкостью, контрастом и восприятием не линейна. Пример: один
сильный артефакт делает снимок диагностически непригодным, даже если
резкость и контраст идеальны. Линейная сумма этого не уловит.

Решение для общего качества:
- **На старте (до первых 500 голосов):** взять напрямую выход
  обученной головы NIMA / CLIP-IQA (одно число, не сумма).
- **После накопления данных:** обучить нелинейный комбайнер —
  малый MLP (2 слоя × 32 нейрона) или градиентный бустинг (LightGBM)
  на признаках = `(μ_sharpness, μ_contrast, μ_artifacts, raw_metrics...)`,
  таргет = pairwise margin loss из людских сравнений по «общему
  качеству».
- Этот же MLP можно уже использовать как production-модель для
  «общего качества», заменив z-score-сумму.

Для остальных критериев (резкость, контраст, артефакты) линейная
комбинация остаётся приемлемой первой версией.

**Normalize** — обсуждается в 3.6.

**Важно:** для каждого из 4 критериев — отдельный набор метрик,
отдельная нормализация, отдельный `μ^(c)`.

### 3.6. Нормализация: z-score или percentile rank

Z-score предполагает приближённую нормальность распределения метрики
по датасету. На медицинских данных это часто не выполняется:
- Энтропия гистограммы может иметь сильную скошенность и тяжёлый хвост.
- NIQE на рентгенах нередко принимает аномально высокие значения по
  сравнению с natural images (потому что и есть «отклонение»).
- CLIP-IQA на однообразном датасете (только один тип снимков) может
  давать почти константные или бимодальные распределения.

В таких случаях лучше **percentile rank normalization**:

```
rank_i = (порядковый номер i по метрике) / N    →  rank ∈ [0, 1]
```

Свойства:
- Распределение становится равномерным U(0, 1) независимо от исходного.
- Робастность к выбросам — один экстремальный снимок не «съедает»
  z-score у всех остальных.
- `Δ` интерпретируется как «разница в перцентилях» — наглядно для
  объяснения порогов оценщикам и в дипломе.

При percentile rank пороги переопределяются:
- `τ_high = 0.30` — изображения отличаются на 30 перцентильных пунктов
  (например, 80-й перцентиль против 50-го).
- `τ_low = 0.10` — отличаются на ≤ 10 пунктов.

Распределение `Δ` для U(0,1) — треугольное, плотность `f(Δ) = 2(1−Δ)`:
- P(Δ ≥ 0.30) ≈ 49%
- P(0.10 ≤ Δ < 0.30) ≈ 32%
- P(Δ < 0.10) ≈ 19%

**Рекомендация:** **percentile rank по умолчанию**. Z-score — только
если пилот (3.0) подтвердил приближённую нормальность распределения.
В коде это разница в 1 строку (`scipy.stats.rankdata` вместо
`(x − mean)/std`), но эффект на устойчивость пороговой фильтрации
существенный.

## 4. Этап 2. Классификация пар по неопределённости

Для каждой пары `(i, j)` и каждого критерия `c`:

```
Δ^(c)(i,j) = |μ_i^(c) − μ_j^(c)|
```

Поскольку `μ` z-нормализованы, `Δ` измеряется в стандартных
отклонениях.

### Три бакета (диапазона)

| Бакет | Условие | Кто решает |
|-------|---------|------------|
| **«Очевидные»** | `Δ ≥ τ_high` | алгоритм автоматически (победитель — изображение с большим `μ`) |
| **«Умеренные»** | `τ_low ≤ Δ < τ_high` | смешанный режим: часть отдаём людям выборочно |
| **«Сомнительные»** | `Δ < τ_low` | **только** люди |

### Стартовые пороги

- `τ_high = 1.5` — разница в 1.5 σ. По нормальному распределению
  это соответствует ситуации «победа очевидна с уверенностью >95%».
- `τ_low = 0.5` — разница меньше 0.5 σ означает «изображения почти
  одинаковы по `μ`» — здесь алгоритм правда не знает.

Эти пороги нужно **откалибровать** по результатам валидационного
эксперимента (Этап 5). Если в «очевидной» зоне согласие людей с
алгоритмом окажется ниже 95% — поднять `τ_high` до 2.0; если в
«сомнительной» зоне согласие при 0.5 уже близко к случайному
(~50%) — `τ_low` можно опустить до 0.3.

### Распределение пар по бакетам

При `μ` ~ N(0, 1) и независимых `i, j`, разница
`X−Y ~ N(0, 2)`, а `Δ = |X−Y|` имеет half-normal распределение со
стандартным отклонением `σ_Δ = √2 ≈ 1.414` и средним `2/√π ≈ 1.13`.
Точные доли (через CDF стандартного нормального `Φ`):

| Бакет | Условие | Доля | Кол-во пар |
|-------|---------|------|------------|
| Очевидные | `Δ ≥ 1.5σ` | `2·(1 − Φ(1.5/√2)) ≈ 28.9%` | ≈ 144 500 |
| Умеренные | `0.5 ≤ Δ < 1.5σ` | ≈ 43.4% | ≈ 217 000 |
| Сомнительные | `Δ < 0.5σ` | `2·Φ(0.5/√2) − 1 ≈ 27.7%` | ≈ 138 500 |

То есть **~29% пар уходят в авто-режим без участия людей** только за
счёт фильтра. Остальное сокращение даёт следующий этап — бюджет.

> [!NOTE]
> При percentile rank нормализации (3.6) распределение `Δ`
> треугольное на [0,1]. Для порогов `τ_high = 0.30`, `τ_low = 0.10`:
> очевидные ≈ 49%, умеренные ≈ 32%, сомнительные ≈ 19%. То есть
> percentile rank потенциально отдаёт ещё больше пар в авто-режим
> (49% против 29%) — это плюс по экономии, но требует более
> аккуратной валидации.

## 5. Этап 3. Бюджет на людские сравнения

Сравнивать все 110 000 «сомнительных» пар не нужно. Известный
результат: для надёжного ранжирования N изображений достаточно
**`O(N log N)` информативных сравнений** (Jamieson & Nowak 2011;
Wauthier, Jordan & Jojic 2013). При N = 1000 это ≈ 10 000 сравнений
на критерий.

### Распределение бюджета B (например, B = 10 000 на критерий)

| Бакет | Доля бюджета | Цель |
|-------|--------------|------|
| Сомнительные | 80% (8 000) | основной информативный сигнал — отсюда мы реально учимся |
| Умеренные | 15% (1 500) | покрытие связности графа сравнений + проверка |
| Очевидные | 5% (500) | санити-чек: отлавливаем случаи, где алгоритм ошибается |

### Внутри бакета — выбор пар по двум приоритетам

1. **Разрежённость графа.** Каждое изображение должно появиться в
   достаточном числе пар (минимум 8–10 раз — иначе BT не сходится по
   нему). Pair-selection отслеживает счётчик `count(i)` и
   приоритезирует изображения с малым счётчиком.
2. **Не-повторяемость.** Уже сравнённую пару не отправляем второй раз
   тому же оценщику.

После накопления первых ~500–1000 человеческих сравнений — переключиться
на **активный выбор пар** на основе уже частично обученной BT-модели:
выбираем пару с максимальной неопределённостью `P(i > j) ≈ 0.5`
(Chen et al. 2013, Crowd-BT).

### 5.5. Связность графа сравнений (защита от распадения BT)

> [!WARNING]
> Это **критический** механизм. Без него весь алгоритм может выдать
> внешне правдоподобный, но математически некорректный финальный
> рейтинг.

**Проблема.** Если все пары с `Δ ≥ τ_high` ушли в авто-режим, то в
человеческом подграфе сравнений возникают изолированные кластеры:
условно «низкое-vs-низкое» и «высокое-vs-высокое», но почти нет
рёбер «низкое-vs-высокое». Тогда BT, построенный только на ручных
голосах (Режим A), не имеет данных, чтобы связать рейтинги двух
кластеров: π-параметры одного кластера могут «плавать» относительно
другого с точностью до общего множителя. λ-сглаживание тянет всё к
среднему и маскирует проблему, но истинной информации между
кластерами нет.

**Диагностика — алгебраическая связность (Fiedler 1973).**

Для графа `G = (V, E)`, где V = изображения, E = ручные сравнения,
строим матрицу Лапласа `L = D − A` (D — матрица степеней, A — матрица
смежности). Второе наименьшее собственное значение `λ₂(L)` — это
**число Фидлера**:
- `λ₂ = 0` ⇔ граф несвязен.
- `λ₂` маленькое (< 0.05) ⇒ граф связан слабо, легко распадается при
  удалении нескольких рёбер — приближение к катастрофе.
- `λ₂` большое ⇒ хорошо перемешан.

Для N = 1000 sparse-вычисление `λ₂` через
`scipy.sparse.linalg.eigsh(L, k=2, which='SM')` занимает < 1 секунды.

**Простая альтернатива (если хочется без матриц):** число компонент
связности через union-find. Если > 1 — граф разбит. O(E·α(N)),
доли секунд.

**Резерв на «мосты» в бюджете.**

Из 10 000 человеческих сравнений на критерий выделяем
**мостовой резерв** 5–10% (500–1 000 сравнений):
- Принудительно генерируем пары, соединяющие **разные квантильные
  кластеры** по `μ` — например, верхний дециль (топ-10% по `μ`) с
  нижним децилем; верхний дециль с медианной зоной и т.д.
- Эти пары попадают именно в «очевидную» зону по `Δ`, но **не**
  отдаются алгоритму — отправляются людям как обычные сравнения.
- Это гарантирует, что в человеческом подграфе всегда будут
  cross-cluster рёбра.

> Это *не* то же самое, что 5%-санити-чек из таблицы бюджета: тот
> сэмплирует «очевидные» пары случайно. Мостовой резерв
> **целенаправленно** сэмплирует те, что соединяют квантильные
> кластеры. Можно объединить в одну категорию «авто-зона / люди»
> (10% бюджета), часть случайная, часть целенаправленная.

**Триггер пополнения.**

После каждой партии в ~500 ручных сравнений пересчитываем `λ₂`. Если
он:
- Падает ниже 0.05 → жёсткий триггер: следующие 200 пар отбираем
  только мостовые.
- Падает между 0.05 и 0.10 → мягкий триггер: добавляем дополнительные
  10% мостовых пар к плановой выборке.

Альтернативный, более прямой триггер: **минимальный cross-decile
охват**. Для каждого изображения `i` проверяем, что среди его ≥ 8
ручных сравнений есть хотя бы один партнёр из противоположного
квантиля по `μ`. Если нет — изображение получает приоритет на
мостовое сравнение.

**Обновлённая таблица распределения бюджета (с мостами):**

| Категория | Доля | Цель |
|-----------|------|------|
| Сомнительные | 75% | основной информативный сигнал |
| Умеренные | 15% | покрытие связности + проверка |
| **Мостовые** (cross-decile) | 7% | **гарантия связности графа** |
| Случайные «очевидные» (санити) | 3% | контроль ошибок метрик |

## 6. Этап 4. Финальное ранжирование Brэдли-Терри

Два режима смешивания авто- и ручных голосов:

### Режим A — «Чистый» (рекомендуется для валидации)

BT строится **только на ручных голосах**. Авто-решения никуда не
записываются. Это даёт честный замер: сходится ли BT-рейтинг с `μ`?

- Плюс: чистая корреляция, никакого «самосбывающегося пророчества».
- Минус: BT не имеет данных по парам в «очевидной» зоне.

### Режим B — «Гибридный» (рекомендуется для production-рейтинга)

`μ` используется как **информативный приор** в итерациях
Зермело-Форда. Алгоритм Брэдли-Терри тогда сходится к смеси
«информации из людей» (где она есть) и «информации из μ» (для пар без
ручных голосов).

**Простой вариант — через инициализацию:**

```
init:   π_i = exp( α · μ_i )         // α = калибровочный коэффициент
repeat: стандартный BT-шаг на ручных голосах
return: π
```

`α` контролирует силу приора:
- `α = 0` → π_i = 1 для всех (нет приора, как в Режиме A).
- `α = 0.5` → для μ ∈ [−2, 2] получаем π ∈ [0.37, 2.7] — умеренный
  приор, который BT может «переубедить» при достаточном числе
  человеческих голосов.
- `α = 1.0` → π ∈ [0.14, 7.4] — сильный приор, BT почти не двигает
  рейтинги без большого числа голосов.

**Рекомендуемое стартовое значение: α = 0.5**, корректировка по
результатам валидации (если ρ между Режимом A и Режимом B растёт с
количеством голосов — приор корректен; если убывает — приор тянет в
сторону).

**Принципиальный вариант — через регуляризацию:**

L2-регуляризованная log-likelihood с привязкой к приору
(Caron & Doucet 2012):

```
L(π) = Σ_pairs log P(observed | π_i, π_j)  −  β · Σ_i (log π_i − μ_i)²
```

`β` — сила регуляризации. При `β = 0` — чистый MLE (Режим A); при
`β → ∞` — π фиксируется на `exp(μ)`. Промежуточные значения дают
компромисс. Это методически корректнее, но требует переписывания
шага BT (нет в текущей реализации
[web/database.php](web/database.php)).

- Плюс: единая структура для всех 1000 изображений.
- Минус: ошибки `μ` просачиваются в финальный рейтинг.

**В коде:** простой вариант (через инициализацию) тривиально
получается из текущего
[web/database.php](web/database.php) → `computeBradleyTerry()`.
Параметризировать `init` через α. Для валидации — Режим A. Для
боевого использования — Режим B со стартовым α = 0.5.

## 7. Этап 5. Валидационный эксперимент

Это **ключ к работе** — здесь проверяется, действительно ли алгоритм
надёжно фильтрует.

### 7.1. Дизайн выборки

Из всех 499 500 пар случайно отбирается **стратифицированная** выборка:

| Страта | Условие | Кол-во пар |
|--------|---------|------------|
| Очевидные | `Δ ≥ 1.5` | 100 |
| Умеренные | `0.5 ≤ Δ < 1.5` | 100 |
| Сомнительные | `Δ < 0.5` | 100 |
| **Всего** | | **300** |

Каждая пара показывается **K = 3–5** разным оценщикам (для оценки
межоценщиковой согласованности и устойчивости).

### 7.2. Метрики качества

**(a) Покритерийное согласие в каждой страте.**

Для каждой страты считается:
```
agreement = доля пар, где >50% оценщиков согласны с предсказанием алгоритма
```

Алгоритм предсказывает: «победитель — у кого больше `μ`». Сравниваем с
большинством людей. Целевые значения:
- Очевидные: ≥ 95% — иначе фильтр опасно агрессивный.
- Умеренные: 70–85% — нормально, тут уже есть неоднозначность.
- Сомнительные: 45–55% — должно быть около случайного (это
  подтверждает, что алгоритм правда не знает, и пары нужно оценивать
  людьми).

**(b) Ранговая корреляция (Spearman ρ).**

Берём финальные BT-рейтинги (Режим A — только на ручных голосах) и
сравниваем с `μ`:

```
ρ = Spearman( π_BT,  μ_pre )
```

ρ варьируется от −1 до +1. Целевое значение: ρ ≥ **0.80** — алгоритм
хорошо угадывает порядок.

**(c) Kendall τ.**

`τ = P(concordant) − P(discordant)` среди всех пар изображений.
Содержательно эквивалентен Spearman, но более устойчив к выбросам.
Целевое: τ ≥ **0.65**.

**(d) Cohen's κ для бинарных решений.**

Для каждой пары: алгоритм даёт «победителя», большинство людей — тоже.
Считаем κ — кореркция на случайное совпадение. κ ≥ **0.6** в
очевидной страте — хорошо.

### 7.3. Калибровка порогов

После валидации:

```
если agreement(очевидные) < 95%:
    повысить τ_high (например, до 2.0 σ)
если agreement(сомнительные) > 60%:
    понизить τ_low (до 0.3 σ) — здесь алгоритм точнее, чем
    предполагалось, можно отдать ему больше пар
```

После калибровки — повторный валидационный замер на новой случайной
выборке.

### 7.4. Покритерийная разница

Корреляция может сильно различаться между критериями:
- «Резкость» — обычно высокая (ρ ≈ 0.85–0.95): дисперсия лапласиана
  напрямую отражает то, что видит человек.
- «Контрастность» — высокая (ρ ≈ 0.80–0.90).
- «Артефакты» — средняя (ρ ≈ 0.60–0.80): NIQE/PIQE обучены на
  естественных, не медицинских изображениях.
- «Общее качество» — самая нестабильная (ρ ≈ 0.50–0.80) без
  fine-tuning.

**Вывод:** возможно, для одних критериев фильтрация будет надёжной с
жёсткими порогами, для других придётся ослабить (`τ_high = 2.5`) или
вообще отключить.

## 9. Псевдокод

```python
# === Этап 1: предоценка ===
from pyiqa import create_metric
import cv2, numpy as np

niqe = create_metric('niqe')
nima = create_metric('nima')

def metric_set(img):
    gray = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)
    return {
        'sharpness':   cv2.Laplacian(gray, cv2.CV_64F).var(),
        'contrast':    gray.std() / (gray.mean() + 1e-6),
        'artifacts':   -niqe(img).item(),     # инверсия: больше = лучше
        'overall':      nima(img).item(),
    }

# собираем по всем 1000 изображениям
features = {path: metric_set(cv2.imread(path)) for path in image_paths}

# z-нормализация по каждому критерию
def zscore_dict(values):
    arr = np.array(list(values.values()))
    mu, sigma = arr.mean(), arr.std()
    return {k: (v - mu) / sigma for k, v in values.items()}

mu = {c: zscore_dict({p: f[c] for p, f in features.items()})
      for c in ['sharpness', 'contrast', 'artifacts', 'overall']}

# === Этап 2: классификация пар ===
TAU_HIGH, TAU_LOW = 1.5, 0.5

def classify_pair(i, j, criterion):
    delta = abs(mu[criterion][i] - mu[criterion][j])
    if delta >= TAU_HIGH:  return 'clear', mu[criterion][i] > mu[criterion][j]
    if delta < TAU_LOW:    return 'uncertain', None
    return 'moderate', None

# === Этап 3: бюджет ===
BUDGET = 10_000
TARGET = {'uncertain': 0.80, 'moderate': 0.15, 'clear': 0.05}

# отбираем пары по приоритетам (см. п.5)
selected = sample_pairs(features, classify_pair, BUDGET, TARGET)

# === Этап 4: запись авто-голосов и отправка людям ===
for (i, j, bucket, auto_winner) in selected:
    if bucket == 'clear':
        record_auto_vote(i, j, winner=auto_winner)
    else:
        push_to_human_queue(i, j, criterion)

# === Этап 5: валидация (после сбора людских голосов) ===
import scipy.stats as st
human_bt = compute_bradley_terry(human_votes_only)
rho, _   = st.spearmanr(list(human_bt.values()), list(mu['sharpness'].values()))
tau, _   = st.kendalltau(list(human_bt.values()), list(mu['sharpness'].values()))
print(f"ρ = {rho:.3f}, τ = {tau:.3f}")
```

## 11. Подводные камни

1. **Domain shift: NR-IQA обучены на естественных изображениях.**
   BRISQUE, NIQE, NIMA — модели тренировались на фотографиях из
   LIVE, TID, CSIQ, AVA-датасетов. Для медицинских снимков
   (рентген, КТ, МРТ, гистология) их статистики не совпадают с
   целевым доменом. Конкретные риски:
   - NIMA может ставить одинаково низкие баллы всем рентгенам
     просто потому, что они монохромные.
   - NIQE штрафует отклонения от статистик «естественных»
     изображений; медицинский снимок по природе является
     отклонением.
   - CLIP-IQA может трактовать диагностически важные особенности
     (опухоль, метаплазию) как «артефакт» и занижать оценку.

   **Защита:** обязательный пилот (3.0) с проверкой ранжирования
   глазами; раннее переключение на fine-tuning (3.4) для
   критерия «общее качество»; полная валидация (Этап 5) до
   масштабного деплоя.

2. **Эффект «самосбывающегося пророчества».** Если в финальный BT
   подмешать авто-голоса с большим весом, то ранжирование сойдётся к
   `μ`, а валидация покажет искусственно высокую корреляцию. Поэтому
   валидация делается на BT, построенном **только** на ручных голосах
   (Режим A).

3. **Распадение графа сравнений (критично).** Если все пары
   `Δ ≥ τ_high` ушли в авто-режим, человеческий граф может разбиться
   на изолированные кластеры по уровню качества — и BT-рейтинги
   разных кластеров перестанут быть сопоставимыми. **Защита** — раздел
   5.5: мониторинг числа Фидлера, мостовой резерв 7% бюджета на
   сравнения cross-decile, триггер пополнения при `λ₂ < 0.05`.

4. **Связность для отдельных изображений.** Если изображение `x`
   попало только в «очевидные» пары и человек его ни разу не сравнивал,
   то в BT по ручным данным оно будет «изолированным» (компенсируется
   λ-сглаживанием в текущей реализации, но качественной информации по
   нему нет). Решение: гарантировать ≥ 5 человеческих сравнений на
   каждое изображение, причём как минимум одно — cross-decile.

5. **Линейность объединения метрик.** Связь между резкостью,
   контрастом и «общим восприятием» нелинейна: один сильный артефакт
   делает снимок непригодным даже при идеальных остальных метриках.
   Линейная сумма z-нормализованных значений это сглаживает. Решение
   (3.5): обучаемый нелинейный комбайнер (MLP / LightGBM) после
   накопления первых ~500 голосов.

6. **Многооценочное усреднение.** Чтобы метрика «согласия» была
   устойчивой, одну валидационную пару должны видеть K ≥ 3 человек —
   иначе случайные шумы одного оценщика искажают корреляцию.

7. **Долгий хвост сходимости BT.** Изображениям с малым числом
   сравнений (даже после стратифицированной выборки) BT присваивает
   рейтинг, близкий к среднему. Это нормально, но при расчёте Spearman
   ρ они «съедают» сигнал. Решение: считать ρ только по изображениям с
   `comparison_count ≥ 5`.

8. **Калибровка под критерий.** Один и тот же `τ_high = 1.5` может
   быть слишком жёстким для «артефактов» и слишком мягким для
   «резкости». Калибровать индивидуально — для каждого критерия свой
   набор `(τ_high, τ_low)`.

9. **Ненормальность распределений метрик.** Z-score предполагает
   приближённую нормальность. Энтропия, выходы NIQE на медицинских
   данных — часто скошены. Решение (3.6): percentile rank
   нормализация по умолчанию; z-score — только если пилот (3.0)
   подтвердил нормальность.

## 12. Ожидаемый итог

| Показатель | Без фильтрации | С фильтрацией |
|------------|----------------|---------------|
| Ручных сравнений на критерий | 499 500 (теор. макс) или ~10 000 (active) | 5 000–10 000 |
| Ручных сравнений по 4 критериям | 40 000+ | 20 000–40 000 |
| Покрытие изображений | 100% | 100% (для авто-зоны через `μ`) |
| Точность ранжирования (Spearman ρ_BT vs ground-truth) | ρ ≈ 0.95 после ~10 k полностью случайных сравнений | ρ ≈ 0.85–0.92, зависит от критерия и качества `μ` |
| Связность графа (`λ₂`) | растёт с числом сравнений | поддерживается ≥ 0.05 за счёт мостовой квоты |
| Время сбора (1 оценщик, 4 сек/пара) | 11+ часов | 6–11 часов |

**Главный выигрыш не столько в количестве сравнений, сколько в том,
что оценщики тратят силы только на информативные пары и не отвечают
на «очевидные» (что снижает усталость и повышает качество субъективных
оценок). При этом мостовая квота даёт математическую гарантию того,
что финальный BT-рейтинг останется численно корректным.**

### Условия применимости

Алгоритм оправдывает себя при выполнении **всех трёх условий**:

1. Калибровочный пилот (3.0) показал, что хотя бы одна метрика для
   каждого ключевого критерия даёт осмысленное ранжирование на
   медицинских данных.
2. Распределение метрик после нормализации не имеет тяжёлых
   мультимодальных хвостов; либо использован percentile rank.
3. Мостовая квота 7% и мониторинг `λ₂` встроены в pipeline сбора
   данных с самого начала.

Если хотя бы одно условие нарушено — фильтрация для соответствующего
критерия отключается (или применяется в более мягком режиме:
`τ_high = 2.5σ`, мостовая квота 15%, обязательный fine-tuning после
первой партии данных).
