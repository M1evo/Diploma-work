# План развития дипломной работы
## «Разработка системы автоматической оценки качества изображений на основе субъективных попарных сравнений»

---

## 1. Оценка текущего состояния

### Что уже работает

Проект — это одностраничное веб-приложение (SPA) на чистом HTML/CSS/JS с PHP-бэкендом. Оценщик вводит никнейм, видит две случайные картинки и выбирает, какая лучше (или «равны»). Результат пишется в текстовый файл `results.log`.

**Сильные стороны:**
- Чистая архитектура: фронтенд и бэкенд разделены через API
- Защита от path traversal при отдаче изображений
- Атомарная запись в лог с блокировкой (`LOCK_EX`)
- Горячие клавиши для быстрой оценки (1/2/3)
- Генератор тестовых данных на Python

**Слабые стороны, которые нужно устранить:**
- Пары выбираются полностью случайно — нет алгоритма
- Хранилище — плоский текстовый файл (нельзя делать запросы, откат, анализ)
- Нет возможности отменить последнюю оценку
- Нет страницы результатов или аналитики
- Один критерий оценки (нет многокритериальности)
- Нет бэкапов

## 2. Объединённый план развития

Ниже — все задачи, сгруппированные по этапам. Каждый этап можно выполнять последовательно: следующий строится на результатах предыдущего.

### Этап 1. Миграция на SQLite

**Зачем:** Текстовый лог `results.log` не позволяет делать выборки, откаты, агрегацию. SQLite — встроенная в PHP БД, не требующая установки отдельного сервера. Один файл `database.sqlite` заменяет лог и даёт все возможности SQL.

**Что делаем:**

1. Проектируем схему БД:

```sql
-- Таблица оценщиков
CREATE TABLE assessors (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    nickname TEXT UNIQUE NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- Таблица критериев оценки
CREATE TABLE criteria (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,        -- например: "Резкость", "Контраст"
    description TEXT                   -- пояснение для оценщика
);

-- Таблица сессий
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    assessor_id INTEGER NOT NULL,
    criterion_id INTEGER NOT NULL,
    started_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    ended_at DATETIME,
    FOREIGN KEY (assessor_id) REFERENCES assessors(id),
    FOREIGN KEY (criterion_id) REFERENCES criteria(id)
);

-- Таблица сравнений (основная)
CREATE TABLE comparisons (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    image_left TEXT NOT NULL,          -- путь к левому изображению
    image_right TEXT NOT NULL,         -- путь к правому изображению
    result TEXT NOT NULL CHECK(result IN ('<', '=', '>')),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    is_undone INTEGER DEFAULT 0,      -- флаг «отменено» (мягкое удаление)
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);
```

2. Создаём PHP-класс `Database` для работы с SQLite (инициализация, миграции)
3. Переписываем `handleVote()` — запись в таблицу `comparisons` вместо файла
4. Переписываем `handlePair()` — подготовка к умному выбору пар (пока оставляем random)
5. Добавляем миграцию существующих данных из `results.log` в SQLite

**Результат:** Все данные хранятся в БД. Старый лог сохраняется как бэкап.

### Этап 2. Отмена последнего действия (Undo)

**Зачем:** Оценщик может случайно нажать не ту кнопку. Без undo это портит данные.

**Что делаем:**

1. **Бэкенд:** Новый API-маршрут `action=undo`
   - Находит последнее сравнение текущего оценщика в текущей сессии, где `is_undone = 0`
   - Ставит ему `is_undone = 1` (мягкое удаление — данные не теряются)
   - Возвращает отменённую пару, чтобы фронтенд мог её показать снова

2. **Фронтенд:**
   - Кнопка «↩ Отменить» в панели управления
   - Горячая клавиша `Ctrl+Z`
   - После отмены: показываем ту же пару, счётчик уменьшается на 1
   - Toast-уведомление «Оценка отменена»

3. **Ограничение:** Можно отменить только последнюю оценку (не цепочку). Это сделано намеренно — если разрешить бесконечные откаты, оценщик может «переиграть» любые решения, что вредит объективности данных.

**Результат:** Оценщик может исправить случайную ошибку одним нажатием.

### Этап 3. Автоматические бэкапы

**Зачем:** Защита от потери данных при сбоях сервера, случайном удалении файла и т.д.

**Стратегия (комбинированная):**

1. **Основная: бэкап каждые N голосов** (рекомендую N = 20)
   - В `handleVote()` после записи проверяем: `SELECT COUNT(*) FROM comparisons` кратен 20?
   - Если да — копируем `database.sqlite` → `backups/backup_2026-04-30_20-15.sqlite`
   - Храним максимум 50 последних бэкапов, старые удаляем

2. **Дополнительная: при уходе оценщика**
   - JavaScript вызывает `navigator.sendBeacon()` на событие `visibilitychange` (вкладка скрыта)
   - Бэкенд `action=end_session` — закрывает сессию (`ended_at`) и делает бэкап
   - Это не гарантировано на 100% (браузер может не отправить), поэтому это дополнение к основному механизму

3. **Формат имён:** `backups/backup_YYYYMMDD_HHMMSS.sqlite`

**Результат:** Всегда есть свежая копия БД, максимум потеря — последние 20 голосов.

### Этап 4. Многокритериальная оценка

**Зачем:** Медицинские изображения нужно оценивать по разным параметрам: резкость, контраст, наличие артефактов, общее диагностическое качество. Один общий критерий «какое лучше» слишком размыт.

**Что делаем:**

1. **Бэкенд:** Предзаполняем таблицу `criteria`:

| ID | Название | Описание |
|----|----------|----------|
| 1 | Общее качество | Какое изображение вы считаете лучшим в целом? |
| 2 | Резкость | На каком изображении детали видны чётче? |
| 3 | Контрастность | На каком изображении лучше различимы структуры? |
| 4 | Артефакты | На каком изображении меньше помех и искажений? |

   Критерии настраиваются — можно добавить специфические для медицинской области.

2. **Фронтенд:** После логина — экран выбора критерия. Оценщик выбирает, по какому критерию сегодня оценивает. Критерий фиксируется на всю сессию (чтобы не путаться).

3. **Результат:** Каждое сравнение привязано к конкретному критерию через сессию. Можно строить отдельные рейтинги по каждому критерию.

### Этап 5. Рейтинговая система (Elo / Bradley-Terry)

**Зачем:** Сырые данные «A лучше B» сами по себе мало говорят. Нужно превратить их в числовой рейтинг каждого изображения, чтобы можно было сказать: «Это изображение — №1 по резкости, а это — №47 по контрасту».

**Два подхода (реализуем оба для сравнения):**

#### 5а. Рейтинг Elo

Та же система, что в шахматах. Каждое изображение начинает с рейтинга 1500. После каждого сравнения рейтинги обновляются:

```
Ожидаемый результат: E_A = 1 / (1 + 10^((R_B - R_A) / 400))
Обновление:          R_A' = R_A + K * (S_A - E_A)
```

Где `K` — коэффициент обучения (обычно 32), `S_A` — фактический результат (1 = победа, 0.5 = ничья, 0 = поражение).

**Плюсы:** Простой, легко реализовать, обновляется инкрементально после каждого голоса.
**Минусы:** Порядок сравнений влияет на результат; нет строгой статистической модели.

#### 5б. Модель Брэдли-Терри

Статистическая модель, которая оценивает «силу» каждого изображения на основе всех собранных сравнений одновременно. Вероятность того, что изображение $i$ победит изображение $j$:

```
P(i > j) = π_i / (π_i + π_j)
```

Параметры `π` оцениваются методом максимального правдоподобия. В отличие от Elo, пересчитывается по всем данным сразу — результат не зависит от порядка.

**Плюсы:** Статистически строгий, даёт доверительные интервалы.
**Минусы:** Нужен пересчёт при добавлении новых данных (но для 1000 изображений — мгновенно).

**Реализация:**
- PHP-скрипт пересчёта рейтингов (запускается при открытии страницы результатов)
- Хранение текущих рейтингов в таблице `ratings`:

```sql
CREATE TABLE ratings (
    image_path TEXT NOT NULL,
    criterion_id INTEGER NOT NULL,
    elo_rating REAL DEFAULT 1500.0,
    bt_score REAL DEFAULT 1.0,       -- параметр π Брэдли-Терри
    comparison_count INTEGER DEFAULT 0,
    PRIMARY KEY (image_path, criterion_id)
);
```

**Результат:** Каждое изображение имеет числовой рейтинг по каждому критерию.

### Этап 6. Умный выбор пар (Swiss Tournament + адаптивный)

**Зачем:** При 1000 изображениях есть ~500 000 уникальных пар. Случайный выбор крайне неэффективен: большинство сравнений будут между изображениями с очевидно разным качеством. Умный алгоритм выбирает «интересные» пары — где рейтинги близки и результат неочевиден.

**Алгоритм:**

1. Берём текущие Elo-рейтинги всех изображений для выбранного критерия
2. Выбираем случайное изображение A (с приоритетом тем, у кого мало сравнений)
3. Ищем изображение B с ближайшим рейтингом к A (но с которым A ещё не сравнивалось или сравнивалось мало раз)
4. Показываем пару (A, B) оценщику

**Зачем это помогает:**
- Сравнения между «похожими по качеству» изображениями дают максимум информации
- Быстрее достигается стабильный рейтинг
- Меньше сравнений нужно для надёжного ранжирования (экономит время оценщиков)

**Результат:** Рейтинг сходится в 3–5 раз быстрее, чем при случайном выборе.

### Этап 7. Страница результатов и визуализация

**Зачем:** Без наглядных результатов проект — просто форма для голосования. Визуализация нужна и для диплома (графики, таблицы), и для понимания прогресса.

**Что показываем:**

1. **Рейтинговая таблица** — топ-N / все изображения, отсортированные по рейтингу
   - Колонки: превью, путь, Elo-рейтинг, BT-score, число сравнений
   - Фильтр по критерию

2. **Матрица сравнений (heatmap)** — квадратная таблица, где ячейка (i, j) показывает, сколько раз i побеждал j
   - Цвет кодирует результат: зелёный = побеждал, красный = проигрывал
   - Показывает покрытие — какие пары ещё не сравнивались

3. **Прогресс оценщиков** — сколько сравнений сделал каждый оценщик, по каким критериям

4. **Экспорт данных:**
   - CSV: `image_left, image_right, result, criterion, assessor, timestamp`
   - JSON: полный дамп для ML-пайплайна
   - Кнопка «Скачать» на странице результатов

### Этап 8. Согласие между оценщиками (IAA)

**Зачем:** Если 10 оценщиков смотрят на одну и ту же пару — все ли они согласны? Если согласие низкое, данным нельзя доверять. Метрики IAA — обязательный элемент любого серьёзного исследования с субъективными оценками.

**Метрики:**

1. **Процент согласия (raw agreement)** — простой процент: в скольки случаях оценщики выбрали одинаковый ответ. Ненадёжен, т.к. не учитывает случайное совпадение.

2. **Каппа Коэна (Cohen's Kappa)** — для двух оценщиков. Корректирует процент согласия с учётом случайного совпадения:
   ```
   κ = (p_observed - p_expected) / (1 - p_expected)
   ```
   - κ > 0.8 — отличное согласие
   - κ 0.6–0.8 — хорошее
   - κ < 0.4 — слабое

3. **Альфа Криппендорфа (Krippendorff's Alpha)** — для произвольного числа оценщиков, работает с неполными данными (не все оценщики оценили все пары).

**Реализация:**
- PHP-скрипт, который считает метрики по данным из `comparisons`
- Отображение на странице результатов с пояснениями
- Это готовый раздел для главы «Результаты» в дипломе

## 4. Список научных источников для основных этапов

1. **Thurstone, L. L.** (1927). A law of comparative judgment. *Psychological Review*, 34(4), 273–286. — Основоположная работа по теории попарных сравнений.

2. **Bradley, R. A., Terry, M. E.** (1952). Rank analysis of incomplete block designs: I. The method of paired comparisons. *Biometrika*, 39(3/4), 324–345. — Модель Брэдли-Терри для вероятностного ранжирования.

3. **Elo, A. E.** (1978). *The Rating of Chessplayers, Past and Present*. Arco Publishing. — Классическая система рейтингования Elo.

4. **Cohen, J.** (1960). A coefficient of agreement for nominal scales. *Educational and Psychological Measurement*, 20(1), 37–46. DOI: 10.1177/001316446002000104 — Каппа Коэна для оценки согласия.

5. **Krippendorff, K.** (2004). *Content Analysis: An Introduction to Its Methodology* (2nd ed.). Sage Publications. — Альфа Криппендорфа для множественных оценщиков.

6. **Chen, X., Bennett, P. N., Collins-Thompson, K., Horvitz, E.** (2013). Pairwise ranking aggregation in a crowdsourced setting. *WSDM '13*. — Модель Crowd-BT: активный выбор пар с учётом надёжности оценщиков.
